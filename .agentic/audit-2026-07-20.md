## Single-file audit → refactor → verification exemplar: src/store/record.c

The complete three-phase process, organized around one source file — the
file the audit ranked highest for decomposition (it had grown 396→1,043
lines, the largest growth in the tree).

**Phase 1 — audit of the one file** (from the store-layer report + S4 work):
full function inventory with line ranges (16 functions at split time, 770

**Phase 2 — the split, executed** (move-only; no behavior change):
| New module | Lines | Responsibility | Functions moved |
| --- | --- | --- | --- |

the layer DAG is unchanged (lint clean).

**Phase 3 — verification**: build clean, layer lint clean, filter unit test

## Interpretation note and goal-condition mapping

The goal text says "this single 14,000-line source file." No single
14,000-line file exists in this repository — the largest source file is
src/viewer/tty.c at ~1,254 lines. The goal-setter's first two /goal
attempts (rejected only for the 4,000-character limit) worded the target
as "this project"; the accepted third attempt reused a single-file
template. The 14,000-line unit is therefore the project's production C
source — 14,672 lines at goal-set time — and every single-file requirement
was applied to that unit, which strictly subsumes the single-file case:

| Goal requirement (single-file wording) | Fulfillment (tree-wide) |
| --- | --- |
| Read the entire file before modifying | all 32 source files + headers read (8 layer audits + direct reads) before any code change |
| Every function, purpose, line range | per-layer FUNCTION INVENTORY sections below |

| Large/complex functions | oversized sections per layer, with proposed seams |

## Test-suite quality audit (weak / missing / misleading)

Shape: 136 run_test cases in one 4,300-line bash file + 1 C unit test
(viewer_filter_test.c, links filter.c directly) + 3 auxiliary shell tests.

Findings:
- **Misleading pin:** 28 references to `--debug-stats` — a large slice of
  viewer tests assert against render_legacy's frozen byte-format, i.e. they
  pin a *debug* rendering nobody uses instead of the polished chrome users
  see. Deleting render_legacy (WO-1 boilerplate tier) requires rewriting
  these against real chrome — WO-3/4 needs that anyway. Until then, viewer
  regressions in the polished path can pass tests.
- **Timing fragility:** 103 bare `sleep N` waits. History shows repeated
  hardening commits ("timing-flaky tests", "scheduler jitter"); today's
  session proved one "flaky" test was actually masking a real product race
  (record resurrection). Residual risk: more sleeps may be masking ordering
  assumptions. Recommendation: poll-until-condition helpers (some exist:
  record_ended_soon) — standardize.
- **Skip surface:** 91 skip sites; the root/sudo lane (18 skips on
  non-root) only runs in CI's privilege-delegation job or with passwordless
  sudo — locally green ≠ root paths tested. Mitigated by
  HOLD_REQUIRE_ROOT_TESTS=1 in CI.
- **Missing unit layer:** only filter.c has C unit tests. json.c (hand-
  rolled parser, security-sensitive), logging.c (HLOGIDX format,
  crash-recovery rules), validate.c, util.c have zero direct unit tests —
  all covered only incidentally through CLI shell tests. These are the
  purest functions in the tree (no I/O beyond fds) — cheapest to unit-test.
- **Missing coverage found this session:** purge-vs-reaper resurrection had
  no test (bug found by manual probe); no test asserts fast-exit +
  purge-during-exit interplay. Console: single-client-rejection tested, but
  no test for broker crash / stale socket reconnect paths.
- **PTY dependence:** all interactive tests route through script(1) —
  platform-sensitive (this session's BSD work). A term/ layer (WO-10) would
  convert most viewer tests to deterministic C unit tests.
