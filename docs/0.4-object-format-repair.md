# 0.4 Object Format Repair Contract

This document locks the repaired storage/inspect shape for Hold runs and the related
profile/grant conversion rules. It supersedes older text that treats profile hashes as
identity, stores duplicated argv0 state, or gives public/private records unrelated shapes.

## Core decision

Hold run storage adopts the Docker-shaped inspect object as the durable run object.

That means:

- storage and inspect use the same shape;
- public/private views are projections of this shape;
- `Path` is the resolved executable;
- `Args` are the arguments after argv0;
- all path-like values are absolute/canonical when the run is created;
- `Config.Origin` is the profile/grant name that created the run;
- top-level `Origin` is the stable 64-hex ID of that profile/grant;
- omit `Origin` and `Config.Origin` for CLI-born runs;
- profile/grant hashes are not object identity;
- sudoers/capability entries pin a derived content hash; normal profile/grant/run objects do not
  use that hash as identity.

## Master run object

This is the target run storage and `hold inspect` shape. Current implementation writes these
Docker-shaped fields alongside legacy lowercase compatibility fields while the shared schema reader is
being consolidated. The private run-record reader now treats the Docker-shaped fields as authoritative
fallbacks for command/state/name/origin/cwd/env/cap/log-driver data when duplicate legacy fields are
absent.

Profile recipe readers also accept the Docker-shaped command subset (`Path`, `Args`, `Config.Env`,
`Config.CapAdd`, `Config.CapDrop`, relevant `Config` terminal fields, and `Config.LogConfig.Type`) so
profile import and stored user/system recipes are parsed through the same compatibility path. Profile
storage/export uses `bin` for the executable and tail-only `args`; legacy full-argv `args`/`argv` remain
accepted on read and are normalized to the in-memory launch vector.

## Field semantics

### Identity and origin

```text
Id             stable 64-hex run ID; display may use first 12 hex
Name           run name; renameable like Docker container names
Origin         stable 64-hex profile/grant ID that created the run; omitted for CLI-born runs
Config.Origin  profile/grant name that created the run; omitted or empty for CLI-born runs
```

`Origin` is not a profile/grant hash. It is the stable ID of the object that birthed the run.

If a profile or grant is renamed later, existing runs keep the origin name captured at birth unless
an explicit lineage-rewrite command is later designed. The ID remains the durable link.

### Command shape

```text
Path  resolved executable path
Args  argv after argv0
```

For example, if the user runs this from `/home/rich/project`:

```sh
hold python3 ./scripts/app.py --config ./dev.toml
```

Hold records:

```json
{
  "Path": "/usr/bin/python3",
  "Args": [
    "/home/rich/project/scripts/app.py",
    "--config",
    "/home/rich/project/dev.toml"
  ],
  "Config": {
    "WorkingDir": "/home/rich/project"
  }
}
```

Do not store the executable twice as both `Path` and `Args[0]`.

At exec time Hold reconstructs:

```text
exec path = Path
exec argv = [Path] + Args
```

### Working directory

`Config.WorkingDir` must be absolute when known. It must not have a trailing slash unless it is `/`.

### Terminal and stdio fields

These fields are durable launch configuration and must not be omitted from full private storage:

```text
AttachStdin
AttachStdout
AttachStderr
Tty
OpenStdin
StdinOnce
```

They describe how Hold launched/captured/attached the run.

### Capabilities

`CapAdd` and `CapDrop` are Docker-shaped launch configuration fields:

```json
"CapAdd": null,
"CapDrop": null
```

or, when configured:

```json
"CapAdd": ["NET_BIND_SERVICE"],
"CapDrop": ["ALL"]
```

They describe Linux capability shaping for the launched host process. They do not imply container
isolation, namespaces, or Docker's complete capability model. Unsupported platforms or unsupported
capability names must fail clearly instead of silently pretending enforcement occurred.

Current 0.4 wiring supports direct run options, saved profile inheritance, explicit user-local
profile creation/import/export, and captive config-shell profile editing:

```text
hold run --cap-drop ALL -- <cmd>
hold profile <name> --cap-drop ALL -- <cmd>
hold profile <name> --cap-add NET_BIND_SERVICE --cap-drop ALL -- <cmd>
hold(config-profile:web)# cap-add NET_BIND_SERVICE
hold(config-profile:web)# cap-drop ALL
```

### Privilege

`Config.Privileged` describes Hold's elevated/root-managed launch path. It does not mean container
isolation exists or was relaxed.

### Logging

`LogPath` and `LogIdx` point to the local raw log and binary sidecar index:

```text
run.log      raw captured stdout/stderr bytes, in capture order
run.log.idx  binary fixed-width index for record offsets, lengths, timestamps, and stream metadata
```

The local hot path is not JSON-lines. Plain `hold logs` reads the raw log for script-friendly output.
The index lets Hold jump by record number and reconstruct timestamps/source metadata without scanning
or parsing JSON.

`Config.LogConfig.Type` defaults to `local`. Future values may include system log destinations such
as journald/syslog, but local raw+idx remains the default storage shape.

#### Local sidecar index v1

```text
Header magic:      HLOGIDX\0
Version:           1
Header size:       80 bytes
Entry size:        16 bytes
Entry count:       number of valid entries
Base timestamp:    Unix epoch microseconds
Raw size/mtime:    sanity info for the indexed raw log

Entry pos_len:     low 44 bits raw byte offset, high 20 bits len_minus_1
Entry time_meta:   low 48 bits timestamp delta in usec, high 16 bits metadata

Metadata bits:     stdout/stderr, no-newline, continuation, truncated, reserved
Capacity:          16 TiB raw offset, 1 MiB record length, ~8.9 years delta/usec
```

## Profiles

Profiles are editable reusable launch definitions. They are the origin analog for profile-born runs.

Profile objects use the same command fields:

```json
{
  "Id": "0711502b4632bc1db9e89a70c4c3d2437fc6b0c9d2699188c4d5f41e05ecae7b",
  "Name": "web",
  "Path": "/usr/bin/python3",
  "Args": ["-m", "http.server", "8080"],
  "Config": {
    "WorkingDir": "/home/rich/project",
    "Env": [],
    "AttachStdin": true,
    "AttachStdout": true,
    "AttachStderr": true,
    "Tty": false,
    "OpenStdin": false,
    "StdinOnce": false,
    "CapAdd": null,
    "CapDrop": null,
    "Privileged": false,
    "LogConfig": {
      "Type": "local",
      "Config": {}
    }
  }
}
```

Profile ID is stable and never changes. Profile name may be renamed. Profile content may be edited.

The private profile object must not contain its own hash. The private profile is the authoritative
source object; its integrity comes from storage ownership, permissions, atomic writes, and the shared
schema reader.

### Public profile projection

The public half of a profile is the bare minimum projection needed for discovery and sudoers/capability
routing. It carries stable identity and name only unless a later command explicitly needs more public
metadata.

Example public profile projection:

```json
{
  "Id": "0711502b4632bc1db9e89a70c4c3d2437fc6b0c9d2699188c4d5f41e05ecae7b",
  "Name": "web"
}
```

Rules:

- `Id` is the stable profile ID.
- `Name` is the public profile/origin name.
- The sudoers/capability hash is computed from canonical private content and written only into the
  sudoers/capability entry that needs to pin privileged execution.
- The hash must not be stored in the private profile or public profile projection.
- Public projection must not expose private command details unless the profile is explicitly public-safe.

## Grants

Grants are subject-scoped authority objects derived from profiles.

Grant path supplies the subject namespace, for example:

```text
/var/lib/hold/grants/users/alice/web-prod.json
```

Grant object:

```json
{
  "Id": "64_hex_grant_id",
  "Name": "web-prod",
  "Origin": "0711502b4632bc1db9e89a70c4c3d2437fc6b0c9d2699188c4d5f41e05ecae7b",
  "Actions": ["start", "stop", "logs", "inspect"],
  "Path": "/usr/local/sbin/web",
  "Args": ["--config", "/etc/web.conf"],
  "Config": {
    "Origin": "web",
    "WorkingDir": "/var/lib/web",
    "Env": [],
    "AttachStdin": false,
    "AttachStdout": true,
    "AttachStderr": true,
    "Tty": false,
    "OpenStdin": false,
    "StdinOnce": false,
    "CapAdd": null,
    "CapDrop": null,
    "Privileged": true,
    "LogConfig": {
      "Type": "local",
      "Config": {}
    }
  }
}
```

Grant rules:

- Grant ID is stable and never changes.
- Grant name is stable and must not be renamed.
- The sudoers/capability hash protects privileged validation, is derived from canonical grant content,
  and is written to the sudoers/capability entry rather than stored as normal object identity.
- The derived hash may change when an admin updates the grant body.
- Runs born from grants set top-level `Origin` to the grant ID and `Config.Origin` to the grant name.
- Subject-private grant material must preserve the profile's execution metadata, including env,
  terminal/detach flags, capability add/drop, restart policy/delay, and log destination.
- The canonical grant digest must cover every preserved execution field that can affect the command,
  privileges, stdio/TTY behavior, restart behavior, or log destination.
- Before writing a sudoers capability, `hold grant` must validate the target profile executable and
  path-like argv values as grant-safe: absolute, resolvable, root-owned through the path chain, and not
  group/world-writable. Unsafe paths fail closed unless the operator explicitly uses `--force`.

Current 0.4 compatibility note: the implemented subject-private grant file is still the existing
`hold.subject-grant.v1` object (`schema`, `subject`, `profile`, `binary_path`, `argv`, optional
metadata fields, `actions`) rather than the final Docker-shaped object above. The repair invariant is
that the same metadata is carried and hashed; broader profile/grant object-reader unification remains
follow-up work.

## Conversion rules

### Run -> profile

Copy these fields:

```text
Path
Args
Config.WorkingDir
Config.Env
terminal/stdio fields
CapAdd/CapDrop
Config.Privileged default if applicable
Config.LogConfig
```

Drop these fields:

```text
run Id
run Name unless explicitly used as requested profile name
State
LogPath
LogIdx
RestartCount
runtime process identity
```

Create a new stable profile ID.

### Profile -> run

Copy profile command/config into the run. Create a new stable run ID. Set:

```text
top-level Origin    = profile Id
Config.Origin       = profile Name
```

### Profile -> grant

Copy profile command/config into the grant. Create a new stable grant ID on create. Preserve grant
ID and grant name on update. The sudoers/capability hash is derived from canonical content, not stored
in the private source profile.

### Grant -> run

Copy grant command/config into the run. Create a new stable run ID. Set:

```text
top-level Origin    = grant Id
Config.Origin       = grant Name
```

The run does not share the grant capability hash as identity.

## Exit status and run supervision

Exit code capture requires a Hold-owned parent for every managed run.

Required process shape:

```text
hold
  run supervisor / console broker
    target process
```

Rules:

- PTY/console runs reuse the console broker as the run supervisor.
- Non-PTY runs get the same lightweight supervisor role without the PTY work.
- The supervisor remains the target's parent and waits on it.
- The supervisor translates wait status into final `State`.
- The supervisor atomically updates the private run object.
- If a public projection exists, the same final state is projected there.

Wait-status mapping:

```text
WIFEXITED(status)    ExitCode = WEXITSTATUS(status)
WIFSIGNALED(status)  ExitCode = 128 + WTERMSIG(status)
```

On normal completion:

```text
State.Status      = exited
State.Running     = false
State.Paused      = false
State.Restarting  = false
State.Dead        = false unless Hold determines a dead/stale condition
State.ExitCode    = mapped exit code
State.FinishedAt  = completion timestamp
```

While active:

```text
State.Status      = running
State.Running     = true
State.ExitCode    = 0
State.FinishedAt  = 0001-01-01T00:00:00Z
```

Do not publish or persist fake exit codes. If a run was created by an older path that did not keep a
Hold-owned parent, Hold may infer that it exited, but the real exit code is unknown.

## Public/private projection

Public files must be projections of this same shape. They may omit sensitive fields but must not use
a different unrelated schema.

Public projections may omit:

```text
Path
Args
Config.Env
Config.WorkingDir
LogPath
LogIdx
runtime process internals when not safe
private owner/provenance
```

Public projections must keep the same `State` object shape as private runs:

```json
"State": {
  "Status": "running",
  "Running": true,
  "Paused": false,
  "Restarting": false,
  "Dead": false,
  "Pid": 1234,
  "Pgid": 1234,
  "Sid": 1234,
  "ExitCode": 0,
  "Error": "",
  "StartedAt": "2026-06-28T17:30:47.559521995Z",
  "FinishedAt": "0001-01-01T00:00:00Z"
}
```

Public projections should keep stable selectors when safe:

```text
Id
Name
Origin
Config.Origin
State
Created
```

Do not create a second public-only state schema. If a value is sensitive, wrong, or unavailable, fix
the producer or omit the entire unsafe projection field by explicit rule; do not silently reshape
`State`.

## User-home sudo routing

If root/sudo starts a command and the executable or any path-like argv resolves inside the invoking
user's home, keep it in that user's local store and chown artifacts back to that user. Do not invent
a global-private run schema for that case.

The path check must cover more than `argv[0]`:

```text
argv path tokens
--flag=/path forms
--flag /path forms where safely inferable
existing relative paths resolved against launch cwd
```

## Shared schema code requirement

Schema parsing and writing must be centralized.

Current duplicated parsing/writing exists across:

```text
src/store/record.c     run records and public run index
src/store/alias.c      user-local profile recipes
src/store/profile.c    system profile bodies
src/access/sudoers.c   subject grant copies and grant hashes
```

Target boundary:

```text
schema layer  parse/write canonical run/profile/grant objects and projections
store layer   choose paths, permissions, atomic temp/fsync/rename
runtime layer create processes and fill State/log/runtime fields
action layer  resolve targets and validate live process state
access layer  sudoers/capability policy and grant hash verification
```

Current implementation progress:

```text
run records       Docker-shaped reader fallback exists
profile recipes   user alias, system profile, JSON import/export share one parser/writer; stored args are tail-only
grants            canonical digest/load reuse shared parser; private grant writer reuses recipe field writer
public index      shares Docker-shaped identity/config/State writer helpers; projection wrapper remains minimal
```

The repair is incomplete if:

```text
- public run JSON is written by a bespoke mini-writer unrelated to full run serialization
- profile/grant/run command fields are parsed by separate incompatible functions
- Config terminal/stdio fields are omitted from full private storage
- Origin/Config.Origin semantics differ between profile-born and grant-born runs
- capability hash is treated as an ID
- Path/Args include duplicated argv0 state
```
