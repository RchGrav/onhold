## Implemented in current working tree, verified locally

### Public and private run state use the same `State` object shape

Decision:

```json
"State": {
  "Status": "running",
  "Running": true,
  "Paused": false,
  "Restarting": false,
  "Dead": false,
  "Pid": 1234,
  "Pgid": 1234,
  "Sid": 1234,
  "ExitCode": 0,
  "Error": "",
  "StartedAt": "2026-06-28T17:30:47.559521995Z",
  "FinishedAt": "0001-01-01T00:00:00Z"
}
```

Current implementation:

- Public run projections now write a Docker-shaped `State` object instead of only a weak hint.
- `state_hint` remains as compatibility/display sugar, but the durable projection includes `Status`,
  booleans, process IDs, exit/error fields, and timestamps.
- Public list reads the projected state instead of hardcoding `unknown`.

### Exit codes are captured by Hold-owned supervisor/reaper paths

Failure state:

- The current record type can store `exit_code`, `term_signal`, and `ended_at`, but normal detached
  runs do not reliably write those values on natural exit.
- Liveness checks can infer exited/stale/running, but cannot recover the real exit code after the
  target has been reaped elsewhere.

Current implementation:

- Non-console detached logger path waits and calls `hold_mark_run_finished`.
- Console broker path waits and calls `hold_mark_run_finished`.
- `hold_mark_run_finished` maps normal exits/signals into Docker-shaped final state and rewrites
  private storage plus the public projection when present.

Wait-status mapping:

```text
WIFEXITED(status)    ExitCode = WEXITSTATUS(status)
WIFSIGNALED(status)  ExitCode = 128 + WTERMSIG(status)
```

### Storage/inspect shape is moving to Docker-shaped and unified

Failure state:

- Run/profile/grant/public/private records had drifted into alien shapes and bespoke readers/writers.
- That makes audit, code review, projection, and grant security harder than necessary.

Implementation requirement:

- Durable run storage and `hold inspect` use the same Docker-shaped object.
- Public files are projections, not separate schema inventions.
- `Path` is the resolved executable.
- `Args` are argv after argv0; do not duplicate executable as both `Path` and `Args[0]`.
- `Config.Origin` is origin name.
- Top-level `Origin` is origin stable 64-hex ID.
- CLI-born runs omit origin fields.

Tracked in:

- `docs/0.4-object-format-repair.md`

Current implementation progress:

- Private run records now include Docker-shaped top-level fields alongside the legacy compatibility fields:
  - `Id`
  - `Created`
  - `Path`
  - `Args`
  - `State`
  - `LogPath`
  - `LogIdx`
  - `Name`
  - `RestartCount`
  - `Config`
- `Path` is written from executable argv0 and `Args` is written as argv after argv0.
- `Config` carries terminal/stdio booleans (`AttachStdin`, `AttachStdout`, `AttachStderr`, `Tty`, `OpenStdin`, `StdinOnce`), env, capabilities, privilege, and log driver metadata.
- `hold inspect` currently wraps the durable record file, so these Docker-shaped fields are now visible there too.
- Legacy lowercase fields remain for compatibility while readers/listing code are still being unified.

### Profile/grant/run identity and hash rules

Decision:

- Run IDs, profile IDs, and grant IDs are stable 64-hex IDs.
- IDs are tracking IDs, not evolving content hashes.
- Profiles are editable and renameable; ID does not change.
- Runs are concrete executions; name can change; ID does not change.
- Grants have stable names and stable IDs; grants are not renameable because logs need a stable alias.
- Profile/private objects do not store their own hash.
- Public profile projection does not store the hash.
- The derived canonical content hash belongs only in sudoers/capability validation entries.

Tracked in:

- `docs/0.4-object-format-repair.md`

### Sudo/user-home routing scans path-like argv, not only target binary

Failure state:

- Earlier behavior checked only the target executable for invoking-user home placement.
- User-home files can also appear in argv path tokens, `--flag=/path`, or `--flag /path`.

Current implementation:

- Root/sudo store selection checks the resolved executable and path-like argv values.
- If any checked path resolves inside the invoking user's home, artifacts stay in that user's local
  store and are chowned back to that user.
- Checked forms now include:
  - argv path tokens
  - `--flag=/path`
  - `--flag /path` where the following token resolves as an existing path
  - existing relative paths resolved against launch cwd

## Logging format implemented in current working tree

### Local logging is raw log plus binary sidecar index

Decision implemented:

```text
run.log      raw captured stdout/stderr bytes, in capture order
run.log.idx  binary HLOGIDX v1 index with offset, length, timestamp delta, and stream metadata
```

This is the format discussed for the fast viewer. It is not JSON-lines and not msgpack. The raw log
keeps plain `hold logs` script-friendly. The sidecar gives the dynamic viewer random access and
timestamp/source metadata without parsing the log body.

Verification evidence from local smoke:

```text
raw log bytes: out\nerr\n
idx header:    48 4c 4f 47 49 44 58 00 01 00 50 00 10 00 00 00
hold logs --plain <id>: printed out and err
record field: log_idx_path present
```

### Linux capability shaping is wired for direct runs and user-local profiles

Current implementation supports direct `hold run` / `hold start` capability options on Linux:

```text
--cap-drop ALL
--cap-drop CAP_NAME
--cap-add CAP_NAME
```

Capability names accept optional `CAP_` prefix and case/hyphen/underscore variants. Unsupported
capability names fail instead of being ignored.

User-local profile inheritance is wired for profiles saved from runs: `profile save` preserves
`cap_add`/`cap_drop` metadata, and profile-born runs carry that metadata into their run record and
launch path.

Explicit user-local profile creation/import/export and captive config-shell editing are also wired:

```text
hold profile <name> --cap-add NET_BIND_SERVICE --cap-drop ALL -- <cmd> [args...]
hold profile export <name>
hold profile export <name> --json
hold profile import <transcript> as <name>

hold(config-profile:web)# cap-add NET_BIND_SERVICE
hold(config-profile:web)# cap-drop ALL
hold(config-profile:web)# no cap-add
hold(config-profile:web)# no cap-drop
```

Verification evidence from local smoke and regression tests:

### Public/private State writer alignment

The public projection now uses the same `State` serializer as private run records. This removes the
separate public-only status/running implementation so the Docker-shaped state object stays identical
for private records and public projections.

The public index reader also parses the projected `State` object and falls back to `state_hint` only for
old public fixtures/records. This lets list/resolve code consume one state shape instead of a separate
public-only mini-state.

This is partial schema unification only: the full public projection writer is still intentionally minimal
and still emits its public-safe wrapper fields separately.

### Console/PTTY exit-state capture

The console broker now records final run state even when PTY master EOF/EIO is observed before the
next non-blocking child wait loop. Before this repair, that break path could leave a console run
recorded as `running` after the target had already exited.

Regression coverage now starts a PTY-backed console run that exits with code 7, attaches through the
console path, and verifies both legacy and Docker-shaped state fields show the retained exit code.

### Protected/system profile metadata preservation

System profile save now preserves the same operational metadata as user-local recipe profiles for the
fields already wired in the current implementation: env, observed ports/volumes metadata, capability
add/drop, restart policy/delay, and log destination. The protected `profiles.json` reader/writer now
round-trips capability and log destination fields instead of silently reducing a root profile to just
binary+argv.

Regression coverage creates a root/system run with `--cap-drop ALL`, saves it as a system profile,
verifies protected `profiles.json` carries the capability metadata, starts through the protected profile
path, and verifies the resulting run record keeps both legacy `cap_drop` and Docker-shaped
`Config.CapDrop`.

### Subject grant execution metadata and canonical digest

Subject-private grant copies now preserve the execution metadata needed to make a grant behave like the
profile it was derived from: env, observed port/volume metadata, capability add/drop, terminal mode
flags, restart policy/delay, and log destination. The canonical grant digest covers those fields, so a
sudoers/capability hash is invalidated if any preserved execution field is removed or changed.

Capability `run <profile> --cap <hash> <token>` requests now carry an internal detach bit for elevated
starts, so sudo-mediated starts preserve the caller's detach/background intent instead of being
reinterpreted as a foreground Docker-style `run` by the root child process.

Regression coverage now verifies a root/system `--cap-drop ALL` profile grant carries `cap_drop` and
`detach`, that the public hash equals the canonical private grant digest, that a grant-born run keeps
the capability metadata, and that removing `cap_drop` invalidates the old capability.

### Docker-shaped WorkingDir projection

Run records already captured the launch cwd under the legacy `observed.cwd` field. The Docker-shaped
`Config.WorkingDir` field now mirrors that absolute cwd instead of emitting an empty placeholder. The
regression test verifies `Config.WorkingDir` matches `observed.cwd` and is normalized as an absolute
path without a trailing slash except `/`.

### Grant path ownership validation

`hold grant` now validates the target system profile's executable and path-like argv values before
writing subject grant material or sudoers entries. Whole-token absolute paths and `--flag=/absolute`
values must resolve through root-owned, non-group/world-writable path components to a regular file or
directory. This extends the old self-binary check to the granted profile inputs instead of only proving
that the `hold` binary is root-owned.

Refusals list the unsafe path reason and print the two intended follow-up shapes:

```text
hold grant --secure <profile> <subject> [actions]   (Recommended after securing files)
hold grant --force  <profile> <subject> [actions]   bypass, warns about weaker security
```

Regression coverage creates a system profile with a user-writable argv script path, verifies the normal
grant refuses without writing private/public grant files, then verifies `--force` is explicit and still
writes the grant material.

### Docker-shaped run-record reader fallback

The run-record writer emits both the Docker-shaped fields (`Path`, `Args`, `State`, `Name`, and
`Config`) and legacy lowercase fields while the schema migration is in progress. The private record
reader now reconstructs runtime state from the Docker-shaped fields when the duplicate legacy fields
are absent:

- `Path` + `Args` rebuild the executable argv used for command display and profile derivation.
- `State.Pid`, `State.Pgid`, `State.Sid`, `State.Status`, `State.ExitCode`, `State.Error`,
  `State.StartedAt`, and `State.FinishedAt` backfill the internal state fields.
- `Name` and `Config.Origin` backfill run name/profile-origin display.
- `Config.WorkingDir` backfills the observed cwd when the legacy `observed` object is absent.
- `Config.Env`, `Config.CapAdd`, `Config.CapDrop`, and `Config.LogConfig.Type` are read as fallbacks.

`Config.LogConfig.Type=local` is treated as the default local raw-log driver and is not persisted back
into profile recipes as an explicit non-default log destination. That avoids poisoning profile saves
with a `local` value in code paths that currently only treat `syslog` as an explicit alternate driver.

Regression coverage now writes a hand-shaped private run record without duplicate legacy `argv`,
`state`, `name`, `alias`, or `observed` fields and verifies `hold ps -a` reads it without a corrupt
record warning, reconstructs the command from `Path`/`Args`, displays `Config.Origin`, and preserves
the run `Name`.

### Shared profile recipe reader

System profile bodies, user-local alias recipes, and profile JSON import now share the same
`hold_parse_profile_recipe_json()` reader instead of maintaining separate parsers for the same command
recipe fields. The shared reader accepts the legacy recipe fields and the Docker-shaped profile/run
fields used by the object-format repair:

- `bin` / `binary_path` / `Path`
- `args` / `argv` / `Path` + `Args`
- `env` / `Config.Env`
- `cap_add` / `CapAdd` / `Config.CapAdd`
- `cap_drop` / `CapDrop` / `Config.CapDrop`
- top-level mode flags, `mode`, and relevant `Config` terminal fields
- `log_destination` / `Config.LogConfig.Type`

As with run records, `Config.LogConfig.Type=local` is treated as the default local raw-log driver and
is not persisted as an explicit non-default destination.

Regression coverage imports a Docker-shaped profile JSON using `Path`, `Args`, and `Config.Env`, starts
the imported profile, and verifies the process sees the configured environment value. Existing profile
hash golden-vector coverage remains green, so the capability hash framing was not changed by the
reader unification.

Run-to-profile conversion (`hold profile save <run> as <name>`) now consumes the already-loaded
`hold_run_record` instead of re-reading raw JSON for legacy lowercase `argv`. That means the same
Docker-shaped run-record fallback is used when saving a profile from a run. The Docker-shaped record
regression now also saves a profile from the hand-shaped record and verifies the derived profile recipe
contains the reconstructed command.

Stored and exported profile recipes now use Docker-style command framing: `bin` stores the executable and
`args` stores only the tail arguments. The shared reader remains backward-compatible with older full-argv
`args`/`argv` arrays by prepending `bin` internally only when the first element is not already the
resolved executable. This keeps on-disk profile data from duplicating argv0 while preserving existing
profile hash/capability behavior after load normalization.

### Shared subject-grant recipe reader

Subject-private grant verification now calls the shared `hold_parse_profile_recipe_json()` reader before
computing the sudoers/capability digest or materializing the granted launch profile. Grant copies still
carry grant-specific `schema` / `subject` / `profile` / `actions` fields, and their writer remains
grant-specific, but command recipe fields are no longer parsed by a separate incompatible grant-only
reader. The grant digest continues to frame the same canonical fields; the parser only normalizes where
those fields can be read from.

Regression coverage now checks that a Docker-shaped subject grant (`Path`, `Args`, `Config`) has the same
canonical digest as the equivalent legacy grant object and can still be used by the elevated capability
run path.

System profile storage, user-local profile storage, and subject-private grant storage now share
`hold_write_profile_recipe_json_members()` for the command recipe fields. The stores still choose their
own outer object/keying model (`hash -> recipe`, `name -> recipe`, or grant subject/profile/actions), but
the actual recipe member list is emitted by one helper so capability metadata, terminal defaults,
restart settings, and log destination do not drift across profile-born and grant-born launches.

Linux capability application now computes the final requested capability set before dropping the bounding
set. This fixes `--cap-drop ALL --cap-add NET_BIND_SERVICE`: the child retains only
`CAP_NET_BIND_SERVICE` instead of losing the add when `ALL` was dropped first. Regression coverage runs a
root-owned child and checks `/proc/self/status` `CapEff` for exactly that bit.

### Docker-shaped public run projection identity

The public run projection still remains a deliberately smaller object than the private run record, but it
now carries the same Docker-shaped identity envelope used by private records: `Id`, `Created`, `Name`,
`Config`, and shared `State`. The writer now uses shared helpers for Docker-shaped identity, redacted
`Config`, and `State` instead of hand-writing those fields in a separate shape. The legacy lowercase
public fields remain for compatibility. The public loader now accepts either lowercase or Docker-shaped
identity keys and can resolve `Origin` from either top-level `Origin` or `Config.Origin`.

Regression coverage verifies that a root/system run's public projection contains the Docker-shaped
identity keys and that public `Id`, `Created`, and every `State` field match the private run record.
