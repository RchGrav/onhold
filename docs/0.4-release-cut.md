## Proposed 0.4.0 cut line

0.4.0 should be the first public drop of the coherent Docker-shaped process-management surface, not the completion of every advanced 0.4 idea. The release should include:

1. Docker-shaped launch/list/log basics:
   - `hold run` foreground by default;
   - `-d/-i/-t/-it/-e/--env-file/--rm/--restart/--name` behavior that is implemented and documented;
   - `hold ps`/`ps -a` Docker-shaped table;
   - JSON local logs with plain/script-friendly decoding;
   - observed ports in `ps`;
   - hard rejections for unsupported Docker flags (`-p`, `-P`, `-v`) rather than fake metadata.
2. The explicit profile namespace:
   - `hold profile <name> ...` for profile creation/update;
   - `hold run <profile>` for profile execution;
   - profile names explicit and never generated;
   - generated run names as container-name-like selectors.
3. Full 64-hex run tracking with 12-hex display selectors.
4. Release documentation that honestly states what is included, what is intentionally unsupported, and what remains in 0.4.x.
5. A green release test suite. Current evidence: `make test` passed with `157 passed, 0 failed, 0 skipped`, plus viewer/hash/version/release-installer targets.

### 0.4.0 must not ship until these gates close

- **Commit split/review:** the current worktree is still too large for a clean release commit. Split or review coherent slices before tagging.
- **Review status refresh:** keep `REVIEW.md` current with exact verification and known caveats.
- **Public docs/changelog:** add a 0.4.0 changelog entry and installation/release notes. Keep `VERSION=0.3.9` until the actual tag/release cut.
- **Grant/security boundary:** either complete the trusted global-profile grant checks or hide/label grant paths as not part of the supported 0.4.0 public surface. Shipping an under-hardened delegation story is riskier than deferring grant polish.
- **Help/parser agreement:** confirm help text, parser behavior, and docs agree for every public 0.4 command and unsupported option.

## 0.4.x backlog after 0.4.0

These items are important, but they do not need to block a coherent 0.4.0 process-management drop if the unsupported parts are honestly documented or hidden.

### Near future before 0.5.0: Dynamic viewer redesign

Implement [Dynamic log viewer design before 0.5](future/viewer-fixes-before-0.5.md) as a separate branch/commit series after the 0.4.0 cut is tested and stabilized:

- keep `hold logs`, `hold logs -f`, and `hold logs -n` plain/script-friendly by default;
- add explicit `hold logs --dynamic` and `hold logs --dyn` full-screen entrypoints;
- add polished 80x24-oriented chrome with right-justified view mode, right-justified `Ctrl-H Help`, header filter field, and centered mode cluster;
- do not add noisy default counters, character counts, file paths, or permanent search indicators;
- implement center-out filtering around the selected record while rendering in log order;
- implement one active visibility bitmap with destructive-until-reset noise exclusion semantics;
- add timestamp/source controls (`Ctrl-T`, `Ctrl-U`, `Ctrl-1..4`, `Ctrl-Y`);
- design Hold-captured logs around a raw/plain log plus durable 16-byte-entry sidecar record map before relying on fast source/timestamp/record-number operations.

### 0.4.x: Grant/global-profile hardening

Finish the trusted execution context work needed for safe delegated execution:

- universal absolute-path normalization for executable and path-like argv entries;
- root-owned/global-profile checks for executables, path-like argv targets, and relevant ancestors;
- personal run/profile classification when target paths live under the invoking user's home;
- refusal reports listing every security issue;
- `--secure` recommended rerun command and `--force` unsafe override wording for grant flows;
- environment/cwd policy for capability hashing and replay;
- tests for symlinks, group/world-writable paths, non-root-owned path args, and home-folder personal-run classification.

### 0.4.x: Captive CLI and profile configuration expansion

Continue the Cisco IOS-style management surface after the normal shell surface is stable:

- richer profile subcommands and contextual help;
- completion and namespace providers for commands, profiles, run IDs, and views;
- constrained custom switches/parameters for granted profiles;
- transcript round-trip coverage for advanced profile options;
- clear redaction/secret-handling policy.

### 0.4.x+: Refinements and scale work

- Dynamic viewer/log-map implementation tracked by the near-future before-0.5 document; do not expand the current 0.4.0 commit with speculative viewer implementation.
- Richer similarity modes beyond Space-to-exclude and Ctrl-R reset.
- Instance allocator syntax such as `8000++` once the profile/schema model is stable.
- Optional logging destinations beyond local JSON and local syslog; use [Logging destinations and capture semantics](future/log-destinations.md) for journald design rules.
- Release/installer hardening beyond the minimum 0.4.0 release gate.
