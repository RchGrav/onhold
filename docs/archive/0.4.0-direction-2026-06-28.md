## 0. Why this session happened

0.4.0 had sprawled ("too much surface") and the core **grant/revoke (sudoers) feature** was in
doubt. The session worked through whether to revert it, landed on a coherent product direction,
audited the security model, and consolidated the scattered repos. Net outcome: **keep delegation,
make the whole CLI faithfully docker-shaped, and lock the execution-context guarantees that make
delegation honestly safe.**

Naming: the tool was renamed `sigmund` → `hold` ("put a process *on hold*" vs `nohup`). Name choice
is not finalized; both appear below.

---

## 1. The governing principle: the inferential trust contract

The CLI mimics docker so a docker user can **predict** behavior from prior knowledge. That yields
two hard rules:

1. **If we expose a verb/flag, it behaves exactly as docker's does.** No "close enough."
2. **If we can't make it faithful, we don't expose it** — ideally reject it loudly so a false
   inference fails fast instead of silently doing the wrong thing.

**Partial mimicry is the one thing we never do** — it's the only option that teaches a wrong
prediction. Every docker surface element sorts into three buckets:

- **Direct analog** — identical flag, identical behavior.
- **Truthful reinterpretation** — same surface, honestly different meaning (documented).
- **Absent** — no substrate for it (isolation / images / registry / daemon), so it's not exposed.

Organizing insight: **hold only owns the "process lifecycle & observation" third of docker.**
Isolation, resources, networking, devices, image build/distribution, and daemon-driven behavior
have no substrate here and are therefore absent.

## 2. Data model

- **profile = image.** Encodes the **full execution context** — argv **+ environment + cwd** —
  not just the command line. Content-hash-pinned for identity/dedup (the old "security via hash"
  framing is retired; the hash is now *identity*, like a docker image digest).
  - **Implication:** the hash must cover **argv + env + cwd**, not just argv (today
    `sigmund_profile_hash_for_argv` hashes argv only). If env/cwd affect execution but aren't
    hashed, behavior can change without the hash noticing.
- **run id = the durable, restartable instance** (docker *container*). Keeps the name **"run id"**
  (we rejected "runner id"), but behaves container-like: `create → Up → Exited → (re)start`, stable
  id across restarts, **logs accumulate across restarts**, persists until `rm`/`prune`. The process
  pid/pgid churns underneath; the run id is stable — exactly docker's container-vs-process split.
- **The run↔profile cycle:** a run can be crystallized into a profile (`profile save <id> as <name>`,
  **never** called `commit` — commit implies filesystem capture we don't do); a profile spawns new
  runs. Profiles are *born out of runs*.
- **Local vs global profiles:**
  - **Global (system-managed)** profiles can back a grant → root execution. All the trusted-path
    rules (§5) apply to them.
  - **Local (user)** profiles run as the user, **can never become a sudoers invocation**, and never
    enter the global store. They are exempt from trusted-path validation (no privilege crossing).
- **Public/private store split** (global runs): a public-readable face exposes only what a granted
  user needs to discover/select (e.g. alias/handle); the **cli, the invoking user, and the logs**
  live in a root-only secured path. Per-verb grants (`tail`/`dump`/`console`) gate access to the
  private data.
- **The `hold shell` adoption feature:** `hold shell` drops you in a real shell; **Ctrl-P Ctrl-Q**
  (docker's detach sequence) "adopts" the live foreground command into a managed background run id.
  Faithful inference: same keys, same outcome ("it keeps running in the background, you get your
  prompt back"). Mechanically the hard part — the shell must run under hold's pty so the command is
  already hold's descendant; adoption = stop forwarding the pty + move to a tracked session + log.
  **Verified working (user-tested 2026-06-28):** `hold shell` performs the *real* process-group
  capture/adoption — it starts a shell under hold's pty and, on the detach-sequence, captures the
  foreground pgid (via `tcgetpgrp` + `/proc/<pid>/{stat,cmdline,exe,cwd}`, normalizing relative
  interpreter argv like `python3 ./server.py`); the demo showed `sleep 30` → `adopted <run id>`.
  Codex's 6/25 `.omx` note calling it a "fake slash-view" is **stale** — it was fixed around
  `75d5bc1` (PTY / controlling-terminal fixes). Lesson: trust the running code over point-in-time notes.

## 3. Docker command surface (buckets)

**Direct analog:** `run`, `ps`, `logs` (alias `tail`), `stop`, `kill`, `rm` (alias `prune`),
`attach`, `inspect` (use this name, not `dump`), `wait`, `version`.

**Truthful reinterpretation / honest analogs (several are genuine gems):**
- `pause`/`unpause` → SIGSTOP / SIGCONT the process group.
- `top` → list processes in the run's group (we track the pgid).
- `port` → observed listening ports (no mapping).
- `rename` → rename the run's NAME; `profile rename` for profiles.
- `stats` → CPU%/MEM/PIDS from `/proc/<pid>/stat` (later; NET/BLK harder).
- `images`/`rmi` → list/remove profiles (profile = image).
- `exec <run id> <cmd>` → spawn `<cmd>` **into the run's process group**, inheriting cwd/env/user,
  with the same `-i`/`-t`/`-d` rules as `run`. Honest divergence (document it): runs on the host
  filesystem, not an isolated rootfs — value is "guarded, alongside this run, in its context," not
  isolation. Its flags are a strict subset of `run`'s (minus `--privileged`).

**Absent (no substrate):** `build`, `bake`, `pull`, `push`, `login`, `logout`, `search`, `commit`,
`cp`, `diff`, `events`, `export`/`import` (container fs tarballs — note the prototype reused these
verbs for "IOS profile transcripts", which is an inference conflict to resolve), `load`, `save`,
`history`, `tag`, `update`, and all of `network`/`volume`/`plugin`/`manifest`/`trust`/`swarm`/`context`/
`builder`/`buildx`. **No `compose`** — it only earns its keep when wiring networks between containers.

**Open rulings (not yet decided):**
- `start`/`create`/`restart` — now have honest analogs given the durable-run-id model (Created
  state, re-start same id, restart = stop+start same id with appended logs). Confirm we implement
  the `Created` state and that `restart` keeps the same id (NOT a new one).
- `-s/--size` on `ps` → reinterpret as **log size** (on-theme: "keeps the log for you"), or absent?
- The **IOS-style captive CLI** (`hold` with no args) + `export`/`import` transcripts: existed only
  to administer the grant gate. With delegation *kept*, it may be rehabilitated as the admin's
  grant-authoring console — but it's the prime "too much surface" candidate and must justify itself.

## 4. `docker run` flags + the default-detach decision

**Honest flags:** `-d/--detach`, `-i`/`-t`/`-it`, `-e/--env` + `--env-file`, `-w/--workdir`,
`--name`, `--rm` (auto-prune record on exit), `--stop-signal`/`--stop-timeout`, `--sig-proxy`,
`--cidfile`, `-a/--attach` + `--detach-keys`, `-l/--label` (+ enables `ps --filter`),
`--ulimit`/`--oom-score-adj` (work without cgroups/root).

**`--init`** — docker's "forward signals + reap" is *literally what hold already is*; make it a
no-op/always-on. **`--restart` (policies)** — the one place faithfulness loses to daemonless: auto-
restart needs a resident watcher; hold is daemonless, so **reject it loudly** (manual
start/stop/restart are fine and need no daemon).

**Everything isolation/resource/network** (`--cap-*`, `--cpu*`, `-m`, `--device*`, `--dns*`,
`--network*`, `-v`, `--mount`, `--tmpfs`, `-p`/`-P`/`--expose`, `--privileged`, `--read-only`,
`--gpus`, `--pid/--ipc/--uts/--userns`, `--security-opt`, `--health-*`, `--platform`, `--runtime`)
→ absent.

**The attach matrix** (the why behind `-i` vs `-t`):

| flags | stdin | pty | meaning |
|---|---|---|---|
| `run` (none) | no | no | foreground, stream stdout/stderr — headless |
| `-i` | yes | no | stdio passthrough (subshell-like) — **the new one** |
| `-t` | no | yes | pty allocated (lie to the program that it's a tty) |
| `-it` | yes | yes | full interactive — **already done** |

`-t` is its own axis because tools branch on `isatty()`. `attach.c` already has a non-pty branch to
lean on for `-i`.

**The default-detach decision (a prototype drift to fix):** the prototype made bare `hold <cmd>`
*foreground*. We want:
- **bare `hold <cmd>` = detached/background "easy mode"** — the deliberate exception, matching the
  original `run_id="$(sigmund …)"` one-liner. Contract-safe because docker has no bare form, so
  there's no inference to violate.
- **`hold run <cmd>` = foreground by default** (docker semantics); `-d` detaches.

**Positional grammar:**

| form | meaning |
|---|---|
| `hold run <profile> [args]` | new run from profile (docker `run IMAGE`); **profile required** |
| `hold <profile>` | easy mode (detached) from a profile |
| `hold ./path` or `/abs/path` | easy mode (detached), forced host path (a `/` ⇒ path, never profile) |
| `hold start/restart <run id>` | reuse an existing run |

`--` is the **literal-command bypass** ("stop parsing as hold flags; the rest is the program + its
args") — used when defining/running a command whose own flags collide with hold's. There is **no
profile-less `run`**; ad-hoc execution is what easy mode is for.

**Other corrections found auditing the prototype `--help`:** `--name` must name the **run** (not a
profile, as the prototype did); `rm` must touch **runs only** (prototype let it remove profiles too —
use `rmi`/`profile rm`); `inspect` is the right verb (keep it).

**`ps` flags:** `-a`, `-q`, `--no-trunc` (full RUN ID + full COMMAND — confirms RUN ID is stored
full, **12 chars is just the display truncation**, widened from 8), `-n`/`-l`, `-f/--filter`
(subset: status/name/id/ancestor→profile/label), `--format` (support `table` + `json`; reject
custom Go templates clearly — we won't embed a template engine). Output mirrors `docker ps -a`
exactly, changing only `CONTAINER ID→RUN ID`, `IMAGE→PROFILE`, and `PORTS` (observed listening
sockets, no `->`). STATUS merges the prototype's separate STATE/RESULT columns into docker's
`Up …` / `Exited (code) … ago` form.

## 5. Security: the delegation model and why it's kept

### The journey
Started wanting to revert grant/revoke (the interpreter hole: you can't contain `node`/`python`).
Corrected mid-session: **the interpreter hole only kills "grant the user a *command* to run."** The
actual design is **"grant a non-root user scoped lifecycle *verbs* over an admin-fixed, hash-pinned
profile"** — the **systemd + polkit pattern**, which is sound. The user controls only the *verb* and
*which run*, never the command/argv.

### Audit verdict (of `src/access/*` in the v0.3.9 tree): **polkit-class, well-built.**
- sudoers rule is regex-anchored: `subject ALL=(root) NOPASSWD: /abs/sigmund ^--system --elevated
  (verbs) [0-9a-f]{8} <alias> <hash>$` — alias and hash are fixed literals; only verb + 8-hex
  selector are user-chosen.
- Elevated executor (`sigmund_cmd_elevated_capability_action`) runs the command **looked up by hash
  from the root-owned system store** — no user argv reaches it; re-validates charset and re-checks
  the alias→hash binding (`sigmund_verify_system_alias_cap`); `start` requires selector `00000000`.
- Defense in depth + atomic sudoers writes (`visudo -cf`, temp+rename), and a self-check that the
  hold binary is root-owned and not group/world-writable.

### The two-risks framing (resolves "was it good work?")
`sudo` has **misconfiguration risk** (awful config language → over-broad grants) and **capability
risk** (what a privileged command can reach). The feature **crushed misconfiguration risk** (hashed,
scoped, auditable, one-command revoke) and **cannot touch capability risk** — and that's fine,
because that's `sudo`'s own posture (GTFOBins = admin misconfig, not sudo CVEs). The only real error
was *implying* containment it can't deliver.

### Responsibility boundary
- **Grantee is contained**; only the **grantor** (root authoring the profile/grant) can foot-gun.
- hold's one real duty: **honest framing** — never make a grant look safer than it is.
- Env-borne injection (`NODE_OPTIONS`, `PYTHONPATH`, `LD_PRELOAD`, …): currently closed only by
  sudo's `env_reset` (start.c: children inherit hold's env). **Better:** the profile owns the env
  explicitly (§2), so the child env is *built from the profile*, making the defense hold's own.

## 6. Trusted-execution-context validator (spec — global profiles only)

Goal: a **verified-immutable execution context** for delegated runs. Validate at grant/profile-create
time; because every validated component ends up root-owned, the grant-time snapshot stays true
(non-root can't alter it later).

**Scope:** GLOBAL/system profiles + grants only. **Local profiles exempt** (no privilege crossing).

**Checks:**
1. **Binary** (`argv[0]`, resolved): root-owned, not group/world-writable. (Already done for the hold
   binary via `validate_sigmund_self_for_sudoers`; extend to the profile binary.)
2. **`/`-trigger path args:** any token containing `/` (and the value side of `--key=/path`) →
   - resolves to an existing object ⇒ enforce root-owned + not other-writable on the object **and
     every ancestor directory** (replacing a file only needs a writable *dir*); sticky-bit aware.
   - doesn't resolve to a path ⇒ treat as a plain string, skip (so `--ratio=1/2`, URLs, regex pass).
   - **symlinks:** `lstat` along the chain (catch writable symlink / root symlink → writable target).
3. **cwd:** never inherit the caller's cwd; the profile pins one (default to a known root-owned dir),
   validated by the same full-chain check. This also closes the bare-relative-name gap (`python
   script.py` resolved against a locked cwd) and implicit-load vectors (sys.path, node_modules, dotfiles).
4. **ACLs / group-writability:** mode bits + owner are necessary but **not sufficient** — a dir can be
   `root:root 0755` yet writable via a POSIX ACL or a group the grantee is in. Check `acl_get_file`
   and group membership, or there's a silent bypass.
5. **env:** the profile's declared env is authoritative and hash-covered (child starts from empty +
   a minimal safe floor + profile vars).

**Honest boundary (document, don't oversell):** this guarantees the **integrity of the execution
context hold constructs** (command, env, cwd, directly-referenced files). It **cannot** vouch for the
**runtime behavior of the trusted binary** — a root-owned config pointing at a writable script, or a
binary that deliberately reads `/tmp/x`. That's inherent to trusting any binary and is the grantor's
domain. Claim **"verified-immutable execution context,"** not "immutable secure runs."

**Net:** profile-owned hash-covered env + enforced root-owned binary + hash-pinned context +
root-immutable cwd + scoped verbs puts hold **meaningfully above hand-rolled sudoers** (which checks
none of these) — honestly, with one named boundary.

## 10. Settled this session — run-id, naming, profile/grant model (authoritative over earlier text)

- **Run id = `sha256(timestamp + cli)`** — computed once, never recomputed or validated; just a unique handle (replaces random). No collision/reserve machinery. Store the full hash, display the first 12 hex (docker-style).
- **Run name = `adjective_noun`** from that same hash: `adj = h % adjectives_count; noun = (h / adjectives_count) % nouns_count`. Compute the counts from the arrays — do NOT hardcode a product constant (`0x682D5` is already off by one vs 1202×355 = 426710).
- **2026-06-29 clarification:** run IDs and any internal profile tracking IDs are one-time creation handles from creation material (`cmd`/normalized `argv`/timestamp/pid or equivalent framed fields). They are not cryptographic signatures and are not rechecked for immutability.
- **Profile name = identity (the pin); profile hash = immutability (security).** For an ungranted profile the hash is just a number, never re-checked. Immutability only becomes load-bearing once a profile is granted.
- **Global eligibility:** none of the command's paths may be in a user's home folder (SPEC §1). **Grant:** every target must be root-owned — a stronger, separate bar. A profile can be global yet not grantable.
- **2026-06-29 clarification:** universal absolute-path normalization comes first. Root ownership is
  only a grant-time security requirement, not a normal/elevated-run routing requirement. If an
  elevated run's executable or any normalized path-like argv target is inside the invoking user's
  home, it becomes an invoking-user personal run ID. Only already-global profiles can be granted;
  granting then checks the executable and every path-like argv target for root ownership and other
  delegated-root safety properties. Refusals should list all reasons and print two follow-up commands:
  the same grant with `--secure` (**Recommended**) and the same grant with `--force` plus a warning
  that it may weaken system security. `--force` cannot bypass absolute normalization, personal/global
  routing, or the requirement that the profile be global before grant.
- **Granting copies the profile into a per-user (private) folder.** The sudoers entry is keyed by **profile name**, carries the grant hash, and is a **machine-generated canonical shape, not a human command** — hold normalizes the user's input and composes the `--cap` argv itself before re-execing under sudo. No `ALL` subject; the weakest grant is a `%group`.
- **Per-user public index** (readable by non-root hold) lists a user's grants (name → hash + param schema) so hold knows to self-elevate and what shape to pass. It never exposes the real command.
- **profile ↔ runid is one-to-many:** `run` mints a fresh unique runid (refuses if one is already active); `--force` starts one more; `--multi N` starts N; `stop <profile>` acts on the single active run or refuses if ambiguous; `--all` acts on all. **Append happens only when you restart a specific stopped runid** — fresh runs never append.
- **Terminology:** `alias` is the deprecated name for `profile`; finishing the alias→profile rename in code/storage is its own cleanup item.
- Decide: `-u/--user` (drop-privs) — support or omit to keep the privilege story clean.
- Decide: `start`/`create`/`restart` exact semantics; `Created` state; `-s`→log-size; IOS CLI fate.
- Decide: which feature branches (if any) to push to GitHub vs leave in the bundle.
- Build: hash-covers-env+cwd; the trusted-context validator; `-i` mode; the docker-`ps` table;
  adjective_noun name generator; revert/cleanup of any access bleed into `list.c` (the
  `<root-managed>` rows) if delegation's UI is reworked.

## 11. 2026-06-29 clarification — `run` vs convenience launch

- `hold run ...` is the Docker-faithful path: foreground by default, `-d`/`--detach` required to background, `-i`/`-t` required for stdin/TTY behavior, and profile names resolve before same-named executables unless `--` forces executable parsing.
- Bare `hold <profile|cmd> ...` preserves the original convenience model: start managed work in the background by default, prefer exact profile matches, and use `--` or a path-like token to force a same-named executable.
- Profile launch-mode defaults (`detach`, `interactive`, `tty`, and legacy `console` compatibility spelling) primarily drive convenience starts, captive CLI starts, grants, and generated transcript/config output. Docker-shaped `run` keeps the current invocation's explicit switches authoritative.
- Add a non-running `profile` creation surface that accepts the same run/profile-shaping switches as `run`, saves them into profile configuration, and emits IOS-style words (`interactive`, `tty`, `detach`, etc.) when exported or edited in captive mode.
- The IOS-style captive CLI is intentionally useful for constrained grants: profile params can expose safe custom switches that Hold validates and maps into canonical argv/env, letting non-root users operate narrow root-managed tasks without arbitrary command control or direct systemd mutation.
