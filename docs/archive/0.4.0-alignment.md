## Alignment matrix

| Area | Priority / release gate | Current branch evidence | Accepted v1 deviation | Follow-up gap/status |
| --- | --- | --- | --- | --- |
| Product identity | P1 release gate | `hold` exists as the operator-facing command surface in branch docs and build examples; On Hold remains project identity. | `hold` remains present for current usage/compatibility while 0.4.0 is not released. | Release notes and install docs must make the final artifact/cutover policy explicit before 0.4.0 release. |
| CLI grammar cutover | P0 release gate | `src/cli.c`/`src/main.c` expose new verbs such as `logs`, `view`, `profile`, `show`, `status`, and `shell` while still mapping through current handlers. | Current branch is an incremental redesign slice, not the full grammar replacement. `show` is acceptable in the target design as a read-only alias/view command, not legacy baggage to remove. | Finish table-driven parser/help agreement, remove or hide legacy primary verbs for 0.4.0, preserve `show` as a non-mutating alias where useful, and document migration in changelog before release. |
| `run` / profile lifecycle semantics | P0 release gate | Spec now separates Docker-shaped `hold run <profile-or-cmd>` from convenience-mode bare `hold <profile-or-cmd>`. It still rejects management forms such as `hold run web stop`; `run` is a start verb, not an execution-record namespace. | Phase 1 intentionally locks the current `hold run -- <cmd>` parser path as launch-only and keeps rejecting `hold run <profile> <management>` while the later Docker-shaped `hold run <profile>` grammar is still unimplemented. Target behavior: `hold run` is foreground unless `-d`; bare `hold` backgrounds by default and may use profile defaults. Profile names beat executable names unless `--` or a path-like token forces executable parsing. | Replace transitional parser/help with target grammar: bare convenience launch, Docker-familiar run flags (`-d/-i/-t/-it/-e/-p/-v/--name/--rm/--restart`), profile `run/stop`, `profile` as a non-running run-switch twin that saves defaults, concrete run-id `stop/kill/logs/inspect/rm`, and tests preventing `run` from becoming a management namespace. |
| Profile model | P0 release gate | Current profile command support uses top-level `hold profiles` for listing and includes run/start/show/export/import plus name-first CRUD/editor commands. The core profile data remains the command argv recipe plus env, metadata, and persistent mode flags. Transcript import/export now emits and consumes Cisco IOS-style `enable`, `configure terminal`, `profile`, `binary`, `argv`, `env`, `interactive`, `tty`, `detach`, `commit`, `end`, `write` commands. | Accepted as profile v1 for the alias-to-profile rename/extension, not the final expanded JSON profile editor. Import/export target is CLI transcript form compiled to/from canonical JSON, not JSON as the primary human interchange. Target transcript grammar uses Cisco IOS-style mode/config commands rather than JSON, Docker flags, or dotted `set` paths. The normal-shell `hold profile ...` shortcut may accept Docker-shaped switches, but it persists their meanings into the profile; exported config should render IOS words. | Continue expanding Cisco IOS-style profile sub-configuration for advanced commands (`alias`, `param`, `multi`, `pty-shim`, `default`), grants, schema validation, redaction/secret handling, `inspect` JSON/object output, shell `profile` creation parity with `run` switches, and broader round-trip transcript guarantees. |
| Captive CLI and completion | P1 release gate | The earlier line-oriented `hold shell`/slash-view prototype has been removed. Ghost text is intentionally out of scope. | Build the Cisco IOS-style captive CLI entered by bare interactive `hold`, reusable CLI completion library, namespace providers for commands/profiles/run IDs/recent commands/views, basic tab completion inside and outside the captive CLI, candidate cycling that includes the original/blank input state, and tests for IOS modes plus contextual help. |
| Run/list/show views | P1 release gate | Current branch still has `show`/`list` compatibility commands and the dynamic log viewer, but no slash-view prompt implementation remains. Run names are deterministic `adjective_noun` values selected from the creation hash by modulo over the compiled wordlist counts. | Replace the legacy random-ID generator with creation-hash ID generation, persist full hashes and generated names, keep 12-hex display truncation, avoid uniqueness retry loops except for fatal full-hash anomaly detection, and update tests/docs around the ID/grant-digest distinction. |

## Release-readiness rule

A 0.4.0 release claim requires all three conditions:

1. `docs/HOLD_0_4_UX_SPEC.md` acceptance criteria are implemented or explicitly narrowed with rationale.
2. This matrix has no open row marked `P0 release gate`, `P1 release gate`, or `P1 release decision` unless the row is explicitly narrowed with rationale before release.
3. Fresh verification, including `make test`, code review, and QA evidence, is recorded in `REVIEW.md` or a release PR.
